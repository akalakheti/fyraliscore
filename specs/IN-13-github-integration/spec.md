# Feature Specification: GitHub Production Integration — App Install, Webhook Ingest, Per-Installation Secrets, Uninstall Chokepoint

**Feature Branch**: `feat/IN-13-github-integration`
**Created**: 2026-05-15
**Status**: Draft
**Input**: User description — "GitHub integration — ingest GitHub events (issues, pull requests, commits, comments, reviews) into the platform, similar in shape to the existing Discord integration. Connect via GitHub App or OAuth, subscribe to webhooks for installed repositories, normalize events into the platform's canonical message/event model, and forward downstream for processing. Should support per-org/per-repo configuration, signature verification, replay protection, and a worker process that consumes the webhook queue."

## Summary

Today, Fyralis can already verify GitHub webhook signatures (`services/webhooks/signatures/github.py` shipped in IN-06), resolve `installation.id → tenant_id` via IN-07's `TenantResolver`, and shape `pull_request` / `push` / `issues` / `issue_comment` / `pull_request_review` / `check_run` payloads into Observations (`services/ingestion/handlers/github.py`). What is **missing** is the entire user-facing onboarding side: there is no `/integrations/github/install` route, no `services/integrations/github/` package, no GitHub App authentication (JWT-from-private-key, installation access tokens), no per-installation webhook secret in the DB-backed secret store, no handling of the `installation` / `installation_repositories` lifecycle events, and no uninstall chokepoint when a customer removes the Fyralis App from their org. The result is that GitHub events can be processed in principle but no customer can self-serve enroll: an operator still has to manually `INSERT INTO provider_installations`, register a webhook secret in env, and pre-share a single secret across every customer (a tenant-isolation regression).

This feature closes that gap. A GitHub org admin clicks "Add Fyralis to GitHub", completes GitHub App installation (selecting which repositories to grant access to), and within seconds `pull_request`, `issues`, `issue_comment`, `pull_request_review`, `push`, and `check_run` deliveries from any selected repository land as Observations under the correct tenant — no Fyralis operator action, no plaintext shared webhook secret, and uninstall events through both the inbound `installation.deleted` webhook and an outbound 401/404 chokepoint correctly disable the row and zero out the per-installation secret.

The shape mirrors IN-08 (Slack) and IN-09 (Discord), but four distinctions are non-negotiable for GitHub and must be preserved through plan and implementation: (a) authentication uses **GitHub App JWTs** (RS256 signed with the App's private key) plus **installation access tokens** (short-lived, refreshed per outbound call), not OAuth refresh tokens; (b) webhook signatures use HMAC-SHA256 with a **per-installation secret** generated by Fyralis at install time and supplied to GitHub via the App configuration (NOT a single shared `GITHUB_WEBHOOK_SECRET` env var, which is the v0 dev-only fallback); (c) uninstall is dual-signal: GitHub **does** emit `installation.deleted` over webhook (unlike Discord), AND outbound calls return 404/401 when access is revoked — both paths must converge on the same idempotent chokepoint; (d) per-installation **repository selection** is set by the GitHub admin during install and changes thereafter via `installation_repositories.added` / `installation_repositories.removed` events — Fyralis MUST honor that selection and drop events from repositories the admin de-selected.

Replay protection is implemented via the existing `(source_channel, external_id)` unique constraint on `observations`, with `external_id` constructed from GitHub's stable node-ids (PR node_id, issue node_id, comment node_id, commit SHA, check_run node_id). The `X-GitHub-Delivery` UUID is also recorded in `content.metadata.delivery_id` for audit but is NOT the dedup key — multiple webhook events about the same logical entity (e.g., a `pull_request.opened` followed by `pull_request.synchronize` for the same PR) carry the same PR node_id but legitimately produce distinct Observations differentiated by action. A separate **delivery_id replay short-cache** rejects the exact same `(installation_id, delivery_id)` pair if seen within 5 minutes — this catches GitHub's documented retry-after-200-but-network-error scenario without conflating with action-level deduplication.

Downstream processing of GitHub-sourced Observations uses the **existing** `post_commit_worker` and `think_worker` paths — there is no new "GitHub queue." The phrase "worker process that consumes the webhook queue" in the input description refers to that existing async pipeline: the webhook endpoint commits the Observation and enqueues a `trigger_queue` row synchronously inside the request; the post-commit worker picks it up and runs downstream actions (LLM passes, recommendations, today-bumps) without blocking the GitHub 200-OK response. No new worker process is introduced by this feature.

Out of scope and explicitly deferred: (a) **OAuth user identity** (GitHub OAuth for users, used to map a GitHub login → Fyralis user) — the GitHub App path is sufficient for org-level webhook ingest; user OAuth becomes relevant when we ship per-user authored comments on Fyralis's side, which is a separate IN-task. (b) **Outbound GitHub REST surface** (commenting on PRs, opening issues, status checks) — the JWT minting and installation-token caching layer required for outbound is in scope at the foundational level (because the uninstall chokepoint needs at least one outbound call site to detect 404), but no product-feature outbound calls are wired. (c) **GraphQL API ingest** — events arrive over webhook (REST-shaped payloads); GraphQL polling is a future enhancement. (d) **Enterprise GitHub (self-hosted GHES) support** — v1 targets github.com only; GHES requires per-installation `api_base_url`, which is a schema addition deferred to a follow-up.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Webhook Deliveries From Installed Repositories Land as Observations (Priority: P1)

As a Fyralis-using team in a GitHub org where the Fyralis GitHub App is installed and has access to a set of selected repositories, every supported event delivered from those repositories must arrive at the gateway as a verified, tenant-resolved request and produce exactly one Observation under the correct tenant within 3 seconds of GitHub's delivery — for `pull_request`, `issues`, `issue_comment`, `pull_request_review`, `push`, and `check_run` events.

**Why this priority**: This is the ingestion contract. The signature verifier and event shapers from IN-06 already exist; without the rest of this feature shipping (install flow, per-installation secrets, installation lifecycle, repo selection enforcement), every customer hits the v0 single-secret env-var path which is a cross-tenant leak risk and breaks the moment a second customer onboards. Every downstream user story (uninstall, repo selection, replay protection, operability) is in service of this contract holding under multi-tenant production conditions.

**Independent Test**: With the Fyralis App installed in a test GitHub org (`provider_installations.installation_id = '12345678'`, `tenant_id` = test tenant) and the per-installation webhook secret stored in the DB-backed secret store, POST a synthetic GitHub `pull_request` event signed with that installation's secret to `/webhooks/github/events`. Assert HTTP 201 (or 200 on dedup), exactly one new row in `observations` with `tenant_id` matching the test tenant, `source_channel='github:webhook'`, `external_id=<PR node_id>`, `source_actor_ref='github:<author_login>'`, and `content.text` shaped per the existing `_shape_pull_request` (`"<author> opened PR #<n> '<title>' against <base_ref>"` for `action=opened`).

**Acceptance Scenarios**:

1. **Given** a GitHub App installation row exists with `installation_id='12345678'`, the per-installation webhook secret is registered in the secret store, and the customer's `selected_repositories` includes `octo-org/octo-repo`, **When** GitHub delivers a `pull_request` `action=opened` webhook signed with that secret for `octo-org/octo-repo`, **Then** the gateway returns 201 with `observation_id` populated, exactly one observation is committed under the test tenant, `external_id` equals the PR's `node_id`, and `content.metadata.delivery_id` records the GitHub `X-GitHub-Delivery` UUID.
2. **Given** the same installation, **When** GitHub redelivers the same `pull_request` event with the same `X-GitHub-Delivery` within 5 minutes (GitHub's documented retry-on-network-error path), **Then** the request is rejected at the replay-cache layer with HTTP 200 (idempotent ack) and zero new observations are committed; the `github_webhook_replay_dropped_total` counter increments.
3. **Given** the same installation, **When** GitHub delivers a `pull_request` `action=synchronize` event AFTER the prior `action=opened` event for the same PR (different `X-GitHub-Delivery`, same PR `node_id`), **Then** the request is processed (NOT dropped as a replay), the existing observation row is upserted in place (idempotent on `external_id` per the existing handler contract), and `content.action='synchronize'` reflects the latest state.
4. **Given** a `push` event with 3 commits on `main`, **When** delivered, **Then** exactly one observation is committed with `external_id='<repo_full>@<after_sha>'`, `content.text="<author> pushed 3 commit(s) to main in <repo>"`, `trust_tier='authoritative'`.
5. **Given** an `issue_comment` `action=created` event with body `"Looks good to me"`, **When** delivered, **Then** the observation's `content.text` contains `"<author> commented on issue #<n> in <repo>: Looks good to me"` (truncated at 200 chars), `trust_tier='inferential'`, and the comment's `node_id` is the `external_id`.
6. **Given** a `pull_request_review` event with `state='approved'`, **When** delivered, **Then** `trust_tier='authoritative'` and `kind='state_change'`. With `state='commented'` or `state='changes_requested'`, `trust_tier='inferential'` and `kind='signal'`.

### User Story 2 — Self-Serve GitHub App Install Without Operator Intervention (Priority: P1)

As a GitHub org admin onboarding to Fyralis, I want to click "Add Fyralis to GitHub" inside the Fyralis app, complete GitHub's App-install consent screen (selecting which repositories to grant access to), and be redirected back to Fyralis with my installation registered and immediately receiving webhooks — no Fyralis operator action required.

**Why this priority**: This is the headline acceptance criterion. Without a self-serve install flow, every new GitHub customer requires manual `INSERT INTO provider_installations`, manual generation and registration of a webhook secret per installation, and manual configuration in GitHub's App settings — the exact anti-pattern IN-08 and IN-09 closed for Slack and Discord respectively. This is the "production posture" half of the task; US1 alone is just receive-side wiring against a hand-seeded row.

**Independent Test**: From a browser session authenticated as a Fyralis tenant, navigate to `GET /integrations/github/install`. Verify (a) HTTP 302 to `https://github.com/apps/<app_slug>/installations/new?state=<token>` carrying a signed state token bound to the authenticated tenant's id (NOT a tenant id from a query parameter). With GitHub's App API mocked, simulate the callback `GET /integrations/github/callback?installation_id=12345678&setup_action=install&state=<token>`. Verify (b) a new `provider_installations` row exists with `provider='github', installation_id='12345678', tenant_id=<authenticated tenant>, enabled=TRUE`, (c) a per-installation webhook secret is generated and stored in `encrypted_secrets` keyed off `secret_ref` and referenced from the installation row, (d) an `installation_audit_log` row with `action='install', status='ok'`, and (e) HTTP 302 to `/integrations/github/installed?installation=<short_hash>`.

**Acceptance Scenarios**:

1. **Given** an authenticated Fyralis tenant clicks the GitHub install button, **When** `/integrations/github/install` is invoked, **Then** the response is a 302 to a GitHub App install URL carrying a signed, single-use state token bound to that tenant's id, with a documented TTL (matching the IN-08/IN-09 state-token TTL).
2. **Given** the user grants the App access to repositories `octo-org/repo-a` and `octo-org/repo-b` and GitHub redirects to `/integrations/github/callback?installation_id=...&setup_action=install&state=...`, **When** the state-token HMAC and expiry verify and the nonce has not been consumed, **Then** Fyralis generates a fresh webhook secret (cryptographically-random, ≥32 bytes), writes it to the secret store with a stable `secret_ref` recorded on the installation row, installs an `installation_audit_log` row with `action='install', status='ok'`, and 302s to a Fyralis success page.
3. **Given** the state token is expired, replayed, or unrecognized, **When** the callback is invoked, **Then** the user is 302'd to `/integrations/github/install-error?reason=state_invalid|state_expired|state_consumed` and no DB writes occur.
4. **Given** the `installation_id` returned by GitHub already maps to a *different* tenant in `provider_installations`, **When** the callback runs, **Then** the response is a 302 to `/integrations/github/install-error?reason=installation_collision`, an `installation_audit_log` row is written with `status='rejected_collision'`, and the conflicting foreign tenant id is **not** leaked in the error response or logs (mirrors IN-08 / IN-09 cross-tenant collision behavior).
5. **Given** the same GitHub org admin re-installs after a prior uninstall, **When** the callback completes, **Then** the existing (disabled) `provider_installations.id` is reused — `enabled` flips back to TRUE, `secret_ref` is rotated to a new value, no duplicate row and no unique-constraint conflict.
6. **Given** an authenticated tenant is mid-install and the callback fails partway through (e.g., secret store write succeeds but installation row insert collides with a freshly created row from a parallel install attempt), **When** the partial failure is observed, **Then** the generated webhook secret is deleted from the secret store before the user-visible error is emitted — no orphaned secret material accumulates across failed installs.

### User Story 3 — Per-Installation Webhook Secret, Not a Shared Env-Var (Priority: P1)

As a security-conscious operator, I need each GitHub App installation to have its own webhook secret generated by Fyralis at install time and stored in the DB-backed encrypted secret store — never a single `GITHUB_WEBHOOK_SECRET` env var shared across every customer.

**Why this priority**: A shared webhook secret means that any customer with the env var (or anyone who exfiltrates it) can forge webhook deliveries for every other tenant on the platform. This is a structural multi-tenancy break that IN-08 already solved for Slack by per-installation signing-secret storage. Letting it slide for GitHub is a security regression masquerading as a v0 simplification. This must ship in v1.

**Independent Test**: Run the OAuth install flow against a mocked GitHub App API for two distinct tenants. Verify (a) two separate rows in `encrypted_secrets` with distinct `secret_ref` values, (b) each `provider_installations` row references its own `secret_ref` and neither row references the other's, (c) the legacy env var `GITHUB_WEBHOOK_SECRET` is not consulted during signature verification when an installation-scoped secret exists (verified by setting the env var to a known-wrong value and confirming a delivery signed with the per-installation secret still verifies). Verify (d) the dev-only env-var fallback at `services/webhooks/signatures/github.py` continues to work when no installation row matches the delivery — required for bootstrap PINGs from GitHub before any install is registered.

**Acceptance Scenarios**:

1. **Given** tenant A and tenant B each complete the GitHub install flow, **When** the install completes, **Then** `encrypted_secrets` holds two distinct rows with distinct ciphertexts, distinct `secret_ref` values, and neither secret value is derivable from the other (cryptographically random).
2. **Given** tenant A's installation, **When** a delivery signed with tenant B's webhook secret is posted to `/webhooks/github/events` carrying tenant A's `installation.id`, **Then** signature verification fails with `signature_mismatch` and HTTP 401; no observation is written.
3. **Given** the `GITHUB_WEBHOOK_SECRET` env var is set to `"v0-fallback"` and a tenant has a per-installation secret in the secret store, **When** GitHub delivers a webhook signed with the per-installation secret, **Then** verification succeeds against the installation-scoped secret and the env var is not consulted in the per-tenant verification path.
4. **Given** the `GITHUB_WEBHOOK_SECRET` env var is set AND no installation row matches the delivery's `installation.id`, **When** the delivery arrives, **Then** the request fails with `unknown_installation` (401) — the env var is NOT used as a fallback secret for unrecognized installations, only for the bootstrap PING path documented in FR-016.
5. **Given** a tenant uninstalls the App and re-installs (US2.5), **When** the re-install completes, **Then** a *new* webhook secret is generated and stored (different from the prior secret); the prior secret is deleted from the secret store. Old deliveries signed with the prior secret are rejected.

### User Story 4 — Uninstall Detection via `installation.deleted` Webhook AND Outbound 404 Chokepoint (Priority: P2)

As a GitHub org admin who removed the Fyralis App from my organization, I expect that the very next signal Fyralis receives — whether GitHub's `installation.deleted` webhook arrives first OR an outbound Fyralis call to the GitHub API returns 404 first — automatically disables my installation, zeroes my webhook secret, and rejects any further webhook deliveries with `unknown_installation`.

**Why this priority**: GitHub emits `installation.deleted` as a webhook on uninstall (this is the easy path), but webhook delivery is not guaranteed: GitHub's docs note "if your application is unreachable when an event occurs, GitHub will retry... but eventually gives up." Without an outbound chokepoint, a delivery missed during planned downtime leaves a `provider_installations.enabled=TRUE` row pointing at a revoked installation forever; subsequent outbound API calls will 404 and silently spam logs without disabling the row. With both paths converging on the same idempotent chokepoint (matching IN-08 Slack uninstall and IN-09 Discord bot-kick), correctness is preserved under either single-signal scenario.

**Independent Test**: (a) Seed an enabled installation. POST a synthetic GitHub `installation.action='deleted'` webhook signed with the installation's secret. Assert: `enabled=FALSE` on the row, secret deleted from the store, `installation_audit_log` row with `action='uninstall', status='ok'`, next inbound delivery for the same installation returns 401 `unknown_installation`. (b) Re-seed the installation. Stub the outbound GitHub REST client (`services/integrations/github/client.py::create_installation_token`) to return 404 on the next attempt. Trigger any outbound (e.g., a token refresh). Assert the identical chokepoint outcome.

**Acceptance Scenarios**:

1. **Given** an enabled installation and a GitHub `installation` webhook with `action='deleted'`, **When** the webhook is verified and dispatched, **Then** no Observation is written (uninstall is a lifecycle event, not signal), the installation row's `enabled` flips to FALSE, the webhook secret is deleted from the secret store, an audit row is appended, and the next `/webhooks/github/events` delivery for that `installation_id` returns 401 `unknown_installation`.
2. **Given** an outbound GitHub API call returning HTTP 404 with `documentation_url` containing `/rest/apps/apps#get-an-installation` (the canonical "installation does not exist" response), **When** the chokepoint observes it, **Then** identical behavior to (1) — disable + zeroize + audit, idempotent on re-entry.
3. **Given** an outbound GitHub API call returning HTTP 401 with `message="Bad credentials"`, **When** the chokepoint observes it, **Then** identical behavior to (1) and (2) — bad credentials on a previously-working installation indicate revocation.
4. **Given** the installation was just disabled, **When** the next inbound webhook for that `installation_id` arrives, **Then** the response is 401 with `unknown_installation`, the `installation_id` is NOT echoed in logs (only short hash), and no Observation is committed.
5. **Given** the inbound `installation.deleted` webhook AND the outbound 404 chokepoint fire concurrently (race during a deploy or operator action), **When** both paths execute, **Then** the outcome is identical to a single-path execution — disable is idempotent, double secret-delete is suppressed with the same pattern as IN-09 (the second `SecretNotFoundError` is caught and logged at INFO), and two audit rows with `action='uninstall', status='ok'` are an accepted tradeoff for lock-free correctness.
6. **Given** an `installation_repositories` webhook with `action='removed'` listing one of the previously-selected repos, **When** dispatched, **Then** the installation row is NOT disabled (this is per-repo, not per-installation), but the selected-repository allowlist is updated and subsequent webhook deliveries from that repo are dropped per US5.

### User Story 5 — Per-Repository Selection Honored on Inbound Deliveries (Priority: P2)

As a GitHub org admin who installed the Fyralis App with access to only repos A and B (not repo C), I expect that webhook deliveries from repo C — which can still reach Fyralis because GitHub sends webhook events for repos in the installed organization but the App must not act on data outside its grant — are dropped silently with a metric, never producing an Observation.

**Why this priority**: GitHub App permissions are explicit and granular; respecting the admin's repository selection is a trust requirement. The platform must not ingest data from a repo the admin de-selected — that is a privacy regression. The selection is mutated post-install via `installation_repositories.added` and `installation_repositories.removed` webhooks; Fyralis must persist the current allowlist and honor it on every delivery.

**Independent Test**: Install the App in a test org with `selected_repositories = ['octo-org/repo-a', 'octo-org/repo-b']`. POST a `pull_request` delivery for `octo-org/repo-c` (signed with the installation's secret, valid `installation.id`). Assert HTTP 202 (or 200) acknowledged-but-dropped, zero observations, and `github_webhook_filtered_repo_total{reason="not_selected"}` increments. Then POST an `installation_repositories.added` webhook adding `octo-org/repo-c`. Then re-POST the `repo-c` `pull_request` delivery. Assert the second attempt produces one observation.

**Acceptance Scenarios**:

1. **Given** an installation with `selected_repositories=['org/a', 'org/b']` and a verified `pull_request` delivery for `org/c`, **When** dispatched, **Then** zero observations are committed, `github_webhook_filtered_repo_total{reason="not_selected"}` increments by 1, and the response is HTTP 200 with `{handled: 'filtered_repo'}` (NOT 401 — the signature was valid, the installation is real, only the repo is outside the grant).
2. **Given** the same installation and a verified delivery for `org/a` (in-selection), **When** dispatched, **Then** exactly one observation is committed per US1.
3. **Given** an `installation_repositories` webhook with `action='added'` adding `org/c`, **When** processed, **Then** the installation row's `selected_repositories` is updated (mechanism TBD in plan — see Assumptions for the storage decision); no observation is committed (lifecycle event).
4. **Given** the same `action='added'` webhook is redelivered, **When** processed, **Then** the allowlist update is idempotent — the repo is not double-counted and no audit row reports a state change beyond the first.
5. **Given** an `installation_repositories` webhook with `action='removed'` listing `org/a`, **When** processed, **Then** the allowlist is updated and subsequent deliveries from `org/a` produce the same dropped-with-metric outcome as (1).
6. **Given** an installation row has no recorded `selected_repositories` (legacy or migration edge), **When** any inbound delivery arrives, **Then** the request is treated as `selected_repositories='all'` (the GitHub App was installed with "All repositories" permission, which omits the explicit list); the delivery is processed normally per US1.

### User Story 6 — Replay Protection Against Re-Delivered Webhooks (Priority: P3)

As the platform, I need to drop GitHub webhook deliveries that arrive with a `X-GitHub-Delivery` UUID Fyralis has already processed within the last 5 minutes — preventing GitHub's at-least-once retry semantics from producing duplicate post-commit work (e.g., a single PR opened event triggering two LLM passes because GitHub retried after a transient 502 from our side).

**Why this priority**: The existing `(source_channel, external_id)` dedup at the observation layer catches duplicates *at the row level* — a re-delivered `pull_request.opened` will produce zero new rows. But the post-commit pipeline (think_worker, recommendations, today-bumps) fires off the INSERT result, not the dedup result; a re-delivered event that hits the upsert path can still trigger the post-commit pipeline a second time. Replay protection at the delivery level (idempotent on the `X-GitHub-Delivery` UUID with a short TTL) is the structural guard. It is P3 because the observability/audit downside of double-firing is bounded and recoverable; correctness still holds. But for a production-grade platform it must ship.

**Independent Test**: POST a verified `pull_request` event with `X-GitHub-Delivery: 11111111-2222-3333-4444-555555555555`. Assert HTTP 201, one observation written, post-commit work enqueued. POST the same payload with the same delivery ID within 60 seconds. Assert HTTP 200 with `{handled: 'replay'}`, zero new observations, zero new post-commit work enqueued, `github_webhook_replay_dropped_total` increments. Wait 5 minutes (or fast-forward the cache clock). POST again. Assert the cache has expired and the request is processed (then immediately deduped at the observation layer).

**Acceptance Scenarios**:

1. **Given** a delivery is processed end-to-end, **When** GitHub re-delivers the same `(installation_id, X-GitHub-Delivery)` within 5 minutes, **Then** the second request returns HTTP 200, zero observations are committed, zero `trigger_queue` rows are enqueued, and `github_webhook_replay_dropped_total{provider="github"}` increments.
2. **Given** the replay cache has expired (TTL > 5 minutes since the first delivery), **When** the same delivery is re-sent, **Then** the request is processed normally; observation-layer dedup catches it and returns 200 with `deduped=true`.
3. **Given** two different deliveries with distinct `X-GitHub-Delivery` UUIDs but the same underlying event payload (e.g., manually replayed via GitHub's UI), **When** both arrive within 5 minutes, **Then** the replay cache does NOT collapse them (different delivery IDs); observation-layer dedup is responsible for the second case.
4. **Given** the replay cache is implemented as an in-process LRU and the process restarts mid-window, **When** the same delivery arrives post-restart, **Then** the request is processed normally — replay-cache state is not durable; observation-layer dedup remains the correctness backstop. This is an accepted property documented in the design.

### User Story 7 — Operational Observability and Per-Installation Metrics (Priority: P3)

As an on-call operator, I need to see the GitHub integration's delivery rate, signature failure rate, replay drop rate, repo-filtered drop rate, and per-installation health via metrics — so that a single tenant's misconfiguration (revoked secret, missed installation event, mis-routed delivery) is visible without grepping logs.

**Why this priority**: The first six user stories define correctness. This one defines operability. A signature failure rate climbing for one installation is the leading indicator of a rotation gone wrong; a sudden spike in `github_webhook_filtered_repo_total` after an `installation_repositories.removed` is the leading indicator that the customer's intent has changed. Without per-label metric series, both signals are buried in volume.

**Independent Test**: Run the integration through a synthetic workload: 100 verified deliveries, 10 signature-fail attempts, 5 replays, 20 repo-filtered drops, 1 `installation.deleted`. Scrape the metrics endpoint. Verify per-counter: `github_webhook_received_total >= 136`, `github_webhook_verified_total >= 100`, `github_webhook_signature_failure_total >= 10`, `github_webhook_replay_dropped_total >= 5`, `github_webhook_filtered_repo_total >= 20`, `github_webhook_lifecycle_total{event="installation.deleted"} >= 1`. Verify a log line for an `installation.deleted` event does NOT contain the raw `installation_id` (short-hash only, mirroring IN-09 SC-006).

**Acceptance Scenarios**:

1. **Given** the gateway has processed a mix of valid, replayed, and repo-filtered deliveries, **When** an operator scrapes `/metrics`, **Then** the counters `github_webhook_received_total`, `github_webhook_verified_total{result}`, `github_webhook_signature_failure_total{reason}`, `github_webhook_replay_dropped_total`, `github_webhook_filtered_repo_total{reason}`, `github_webhook_lifecycle_total{event}` are all observable and non-decreasing.
2. **Given** a webhook fails signature verification, **When** the failure is logged, **Then** the log line contains `installation_id_hash` (short hash), `delivery_id`, `event_type` from `X-GitHub-Event`, and `reason` — but never the raw `installation_id` or the candidate signature value (FR-016 parity with IN-06).
3. **Given** an `installation` lifecycle webhook (install / delete / suspend / unsuspend), **When** dispatched, **Then** the line contains `installation_row_id`, `tenant_id`, and `action`, but never the raw `installation_id` or `account.login`.
4. **Given** the outbound GitHub REST client mints an installation access token, **When** the mint succeeds, **Then** a metric `github_installation_token_mint_total{result="ok"}` increments and the token itself is never logged in plaintext. On failure, `result="error"` and the GitHub error code label.

### Edge Cases

- **Bootstrap PING before any install**: GitHub sends a `ping` event when a webhook endpoint is first configured. The current verifier supports an env-var fallback secret for the dev-bootstrap path; this MUST continue working so an operator can register the App's webhook endpoint before any customer installs. PINGs do NOT produce an Observation.
- **App private key rotation**: Operators may rotate the GitHub App's private key. The integration MUST read the private key from a single secret source (e.g., env var or secret store entry referenced by `GITHUB_APP_PRIVATE_KEY_REF`); rotation is an operator action that takes effect on the next JWT mint. No restart required if reading from the secret store; an env-var path requires a deploy. This is an operator-side concern; the integration only needs to NOT cache the private key across requests in a way that prevents rotation.
- **Webhook delivery for a repository in a not-yet-known installation**: GitHub can deliver `repository` events (e.g., `repository.created`) before the `installation.created` event arrives if events fan out in parallel. The current verifier already returns `unknown_installation` (401) in this case; GitHub retries; eventually `installation.created` arrives and the next retry succeeds. This is an accepted property — no special handling required, but the `github_webhook_signature_failure_total{reason="unknown_installation"}` counter MUST distinguish this from a true signature mismatch.
- **Very large webhook payload (e.g., `push` event with hundreds of commits)**: The existing `MAX_PAYLOAD_BYTES=1MB` cap from IN-01 applies; deliveries above the cap return 413. GitHub does not retry 413s. The `content.text` shape ("X pushed N commits to branch") does not scale with N; the full commit list is captured in `content.raw_payload` if and only if the payload is under the cap.
- **Unsupported event type (e.g., `release`, `discussion`)**: The existing handler returns `ValidationError(unsupported github event type)` which the router maps to HTTP 400. We do NOT silently drop unsupported events — GitHub will retry; the operator's signal is the 400 rate and the `supported` list in the error context. This is by design: silent ignore would hide a missed event-type wiring.
- **`installation.suspend` / `installation.unsuspend`**: GitHub allows orgs to temporarily suspend an installation without uninstalling. On `installation.suspend`, the row's `enabled` flips to FALSE (same as `deleted`) and the secret is NOT deleted. On `installation.unsuspend`, `enabled` flips back to TRUE and the existing secret is reused. This is distinct from `installation.deleted` (which is permanent and DOES delete the secret). The `installation_audit_log` records each transition.
- **`installation_repositories.added` for a repo the installation didn't grant**: Defensive — Fyralis trusts GitHub's payload; if GitHub says a repo was added, it was. No additional verification call against GitHub's API is made (would add latency for no security gain).
- **Multiple Fyralis App installations within the same GitHub org (rare but possible if the org installed twice)**: Each install gets its own `installation_id` and its own row; both can coexist. Deliveries route via `installation.id` so no cross-talk. The collision check in US2.4 fires only when the same `installation_id` would rebind to a different tenant — re-installing to the same tenant is US2.5 (re-install).
- **Webhook delivered during a deploy cutover**: If the new pod's secret store has been wiped or the installation hasn't been migrated, the verifier returns `signature_mismatch` and GitHub retries within 1 hour. This is the same property as IN-08; the runbook covers it.
- **`X-GitHub-Delivery` header missing or non-UUID**: Defensive — the replay cache short-circuits the lookup (cache key is unconstructable) and the delivery is processed normally. Observation-layer dedup remains the correctness backstop. A WARN log is emitted with `reason='missing_delivery_header'` so operators see anomalous traffic.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST expose `GET /integrations/github/install` and `GET /integrations/github/callback` mounted on the integrations router. The install route MUST require an authenticated Fyralis tenant session (Bearer auth via the gateway's existing path-allowlist exclusion for `/integrations/*`); the callback route is public, authenticated by the state-token HMAC + nonce-consume pattern shared with IN-08 / IN-09.
- **FR-002**: On `GET /integrations/github/install` for an authenticated tenant `T`, the system MUST issue a single-use state token bound to `T.id` (NOT a query-parameter tenant id), persist the nonce + expiry in `oauth_install_states`, and 302 the user to `https://github.com/apps/<app_slug>/installations/new?state=<token>`. State token TTL, signing-key handling, and nonce-consume semantics MUST match IN-08's existing implementation.
- **FR-003**: On `GET /integrations/github/callback?installation_id=<id>&setup_action=<install|update>&state=<token>`, the system MUST: (a) verify the state token (HMAC + expiry + atomic nonce consume); (b) extract `installation_id`; (c) generate a cryptographically-random ≥32-byte webhook secret; (d) store the secret in the DB-backed secret store with a stable `secret_ref`; (e) UPSERT `provider_installations` with `provider='github', installation_id=<id>, tenant_id=T.id, secret_ref=<ref>, enabled=TRUE`; (f) write an `installation_audit_log` row with `action='install', status='ok'`; (g) 302 the user to `/integrations/github/installed?installation=<short_hash>`.
- **FR-004**: On `setup_action='update'` (the admin changed repository selection mid-install), the callback MUST refresh `selected_repositories` via a single GitHub REST call (`GET /installation/repositories` authenticated with a freshly-minted installation access token) and persist the list. No new webhook secret is generated; no audit row beyond `action='update', status='ok'`.
- **FR-005**: Cross-tenant collisions in the callback MUST 302 to `/integrations/github/install-error?reason=installation_collision` and write an audit row with `status='rejected_collision'`. The foreign tenant id MUST NOT appear in the response body, response headers, or any log line.
- **FR-006**: Re-install for an `installation_id` previously associated with the same tenant MUST reuse the existing row id, flip `enabled=TRUE`, rotate the webhook secret (new random value, new ref), delete the prior secret ref atomically with the rotation, and write an audit row with `action='reinstall', status='ok'`. The unique constraint on `(provider, installation_id)` MUST hold.
- **FR-007**: The `/webhooks/github/events` route (the existing router under `services/webhooks/router.py`) MUST verify HMAC-SHA256 signatures against the per-installation secret loaded via the existing `load_secrets(provider='github', tenant_id, …)` path. The dev-only env-var fallback `GITHUB_WEBHOOK_SECRET` MUST be consulted only when no installation row matches (bootstrap PING) AND when `FYRALIS_ENV != 'prod'`.
- **FR-008**: For every verified, tenant-resolved delivery, the system MUST: (a) check the replay cache keyed on `(installation_id, X-GitHub-Delivery)` and short-circuit with HTTP 200 + `handled='replay'` if a hit within 5 minutes; (b) check `selected_repositories` and short-circuit with HTTP 200 + `handled='filtered_repo'` if the event's `repository.full_name` is not in the allowlist (`selected_repositories='all'` skips this check); (c) on lifecycle events (`installation`, `installation_repositories`), route to the GitHub uninstall/lifecycle handler instead of ingestion; (d) otherwise, call the existing ingestion path via `services/ingestion/handlers/github.py`.
- **FR-009**: The system MUST handle `installation` webhook events with `action ∈ {created, deleted, suspend, unsuspend}` via a new `services/integrations/github/uninstall.py` mirroring the IN-08 Slack uninstall pattern. `deleted`: disable row, delete secret. `suspend`: disable row, retain secret. `unsuspend`: enable row. `created`: no-op (the OAuth callback already wrote the row; lifecycle webhook arrives after and is redundant). All four actions write an `installation_audit_log` row.
- **FR-010**: The system MUST handle `installation_repositories` webhook events with `action ∈ {added, removed}` by mutating the installation row's `selected_repositories` field. Idempotent on the (installation_row_id, repo_full_name) pair — adding a repo already present is a no-op, removing a repo already absent is a no-op. No audit row per change (high frequency, low operator value); a single `installation_audit_log` per webhook arrival with `action='repo_change', status='ok'` and `context.{added, removed}` lists.
- **FR-011**: The outbound GitHub REST client at `services/integrations/github/client.py` MUST mint installation access tokens via the App-JWT flow on demand: (a) sign a 10-minute JWT (RS256) with the App's private key, (b) POST it to `https://api.github.com/app/installations/<installation_id>/access_tokens`, (c) cache the resulting installation access token for the lifetime granted by GitHub (typically 1 hour). The cache key is `installation_id`; the cache MUST be process-local (not Redis-backed in v1, matching the IN-09 token-cache posture).
- **FR-012**: The outbound client MUST observe a uninstall chokepoint identical in shape to `services/integrations/discord/uninstall.py::_disable_and_zeroize_discord`: on HTTP 404 with `documentation_url` ending in `/apps/apps` or `/apps/installations` OR HTTP 401 with `message=Bad credentials`, the chokepoint MUST disable the installation row, delete the webhook secret from the secret store, write an `installation_audit_log` row with `action='uninstall', status='ok'`, and re-raise the underlying `GithubApiError` so the caller sees a consistent error.
- **FR-013**: The inbound `installation.deleted` webhook handler AND the outbound chokepoint MUST converge on the same private function (`_disable_and_zeroize_github`). Concurrent execution MUST be idempotent on the row id: the second `UPDATE provider_installations SET enabled=FALSE` is a benign no-op; the second `secret_store.delete()` raises `SecretNotFoundError` which is caught and logged at INFO; two `installation_audit_log` rows are accepted as the cost of lock-free correctness.
- **FR-014**: The replay cache MUST be an in-process LRU with a 5-minute TTL, keyed on `(installation_id, X-GitHub-Delivery)`. Max entries ≥ 4096. Cache misses bypass the cache and process normally. Cache failures (any internal exception) MUST be swallowed-and-logged with metric `github_webhook_replay_cache_bypass_total` and the request proceeds — replay protection is a defense-in-depth layer, never a correctness gate; observation-layer dedup is the backstop.
- **FR-015**: The system MUST extend the existing `services/ingestion/handlers/github.py::_EVENT_SHAPERS` zero or minimally. Specifically, lifecycle events (`installation`, `installation_repositories`) MUST NOT pass through the shaper registry — they are intercepted at the router layer (FR-008c) and routed to the uninstall handler. The shaper registry continues to hold `pull_request`, `push`, `issues`, `issue_comment`, `pull_request_review`, `check_run` and no additions in v1.
- **FR-016**: Structured log lines emitted by the GitHub integration (router, oauth, uninstall, client) MUST NOT contain raw `installation_id`, `account.login`, `account.id`, or the App's private key material. Acceptable identifiers: `installation_row_id` (Fyralis UUID), `tenant_id`, `delivery_id`, `event_type`, `repo_full_name` (already in the public webhook payload — not sensitive), `installation_id_hash` (BLAKE2b 8-byte digest matching IN-09's `short_guild_hash` construction).
- **FR-017**: The system MUST emit the following metrics: `github_webhook_received_total` (counter), `github_webhook_verified_total{result}` (counter), `github_webhook_signature_failure_total{reason}` (counter, reasons include `signature_mismatch`, `malformed_signature_header`, `missing_signature`, `unknown_installation`, `secret_not_configured`), `github_webhook_replay_dropped_total` (counter), `github_webhook_filtered_repo_total{reason}` (counter), `github_webhook_lifecycle_total{event,action}` (counter), `github_installation_token_mint_total{result}` (counter), `github_install_callback_total{outcome}` (counter, outcomes: `ok`, `state_invalid`, `state_expired`, `state_consumed`, `installation_collision`, `secret_store_error`).
- **FR-018**: The system MUST NOT add a new top-level database table. Persistence reuses the existing `provider_installations` (existing row shape; `selected_repositories` is added as a JSONB column via the same migration that ships this feature OR stored in `provider_installations.metadata` if such a column exists — plan must verify), `encrypted_secrets`, `installation_audit_log`, and `oauth_install_states` tables. If `selected_repositories` requires a new column, the migration MUST be a single `ALTER TABLE` adding a nullable JSONB column with a default of `NULL` (meaning "all repositories").
- **FR-019**: The system MUST NOT modify the shape of `services/integrations/slack/` or `services/integrations/discord/`. New code lives in `services/integrations/github/` (new package: `__init__.py`, `oauth.py`, `client.py`, `uninstall.py`, `metrics.py`, `tests/`). The existing `services/webhooks/router.py` is modified minimally: add the GitHub lifecycle event routing branch parallel to the existing Slack lifecycle branch, add the replay-cache lookup, add the repo-filter check. No other provider's behavior changes.
- **FR-020**: The system MUST NOT consult `GITHUB_WEBHOOK_SECRET` env var in the per-installation verification path under `FYRALIS_ENV=prod`. The env var is dev-bootstrap only; production deliveries MUST verify against per-installation secrets loaded from the secret store.
- **FR-021**: The GitHub App's webhook URL MUST be `https://<gateway-host>/webhooks/github/events`. This is an operator-side configuration in GitHub's App settings, not a runtime concern, but the install runbook MUST document the value.
- **FR-022**: The system MUST handle the GitHub `ping` event (HTTP 200, no Observation) at the router layer BEFORE the unknown-installation check — pings from the bootstrap PING path do not yet have an installation row, and the dev-env-var fallback secret is the only thing verifying them in that pre-install window.

### Key Entities

- **GitHub App Installation**: A persistent association between a GitHub organization (or user) and the Fyralis App. Represented by a row in `provider_installations` (`provider='github'`, `installation_id=<GitHub-issued numeric id as string>`, `tenant_id`, `secret_ref`, `enabled`, plus a new or existing JSONB column holding `selected_repositories`). Created on `setup_action='install'` callback success, mutated on `setup_action='update'`, disabled on uninstall.
- **Installation Webhook Secret**: A cryptographically-random ≥32-byte secret generated by Fyralis at install time, stored in the DB-backed `encrypted_secrets` store, and supplied to GitHub via the App's webhook configuration (the App-level secret in GitHub is a single value; per-installation secrets are achieved by Fyralis generating the secret and updating GitHub's App config via the API at install time — OR by accepting a single App-level secret and per-installation pepper, plan must decide; see Assumptions).
- **GitHub App Private Key**: A PEM-encoded RSA private key used to sign JWTs (RS256) for the `GET /app/installations/...` flow. Stored as a single env var or secret-store entry, NOT per-installation. Read on every JWT mint; never cached in memory across requests in a way that prevents rotation.
- **Installation Access Token**: A short-lived (≤1 hour) token GitHub issues in exchange for a fresh App JWT. Cached in-process per `installation_id` for the lifetime granted by GitHub. Used by every outbound REST call. NOT persisted.
- **Repository Allowlist (`selected_repositories`)**: Per-installation list of `<owner>/<repo>` full names the installation has been granted access to. `NULL` or `'all'` means the customer granted "all repositories" (an explicit GitHub App permission); a list means an explicit selection. Mutated by `installation_repositories.added` / `installation_repositories.removed` webhooks.
- **Replay Cache**: An in-process LRU mapping `(installation_id, X-GitHub-Delivery UUID)` → timestamp, with 5-minute TTL. Not persisted across restarts; observation-layer `(source_channel, external_id)` dedup remains the correctness backstop.
- **Installation Audit Log Entry**: A row in `installation_audit_log` recording `install`, `reinstall`, `update`, `uninstall`, `suspend`, `unsuspend`, `repo_change`, `rejected_collision` transitions for a given installation row. Identical shape to the IN-08 / IN-09 audit log rows.
- **Short Installation Hash**: An 8-byte BLAKE2b digest of `installation_id` used in log lines to disambiguate installations without exposing the GitHub-side enumerable id. Same construction as IN-09's `short_guild_hash`.

## Assumptions

- **GitHub App, not OAuth User**: The user input mentions "GitHub App or OAuth"; v1 implements the GitHub App path because it is the canonical way to receive webhooks for installed repositories and supports app-level authentication that scales across customer orgs. User-level OAuth (mapping a GitHub login → Fyralis user) is out of scope (see Summary).
- **Per-installation webhook secret via App-level config + per-installation pepper, OR via dynamic per-installation secret**: GitHub App configuration historically allowed a single App-level webhook secret. Recent GitHub Apps (and Enterprise variants) support per-installation secrets via the API. The plan phase MUST verify which option github.com currently supports for our App and choose accordingly; this spec assumes the API path exists. If only the App-level secret is available, the implementation falls back to a single secret PLUS a per-installation HMAC pepper computed from `installation_id || tenant_id` and rotated via the install/reinstall flow — the structural security property (cross-tenant forgery requires knowing the pepper AND the App secret) is preserved.
- **`selected_repositories` stored as JSONB column on `provider_installations`**: The simplest persistence path. If `provider_installations` already has a `metadata` JSONB column the list goes inside that; otherwise the migration adds a dedicated `selected_repositories` JSONB column. The plan resolves which.
- **GitHub App private key is operator-supplied**: At App registration time (an operator step done once per Fyralis deployment), GitHub generates the App's private key as a `.pem` file. The operator places it in the secret store under a known ref OR in an env var `GITHUB_APP_PRIVATE_KEY` (multi-line PEM). The integration reads it on demand; rotation is operator-driven.
- **GitHub App slug, app_id, and webhook endpoint URL are operator-configured constants**: Read from env vars (`GITHUB_APP_SLUG`, `GITHUB_APP_ID`) at gateway startup. Documented in the install runbook.
- **`installation_audit_log` and `oauth_install_states` tables already exist**: Shipped in IN-08 (Slack) and reused by IN-09 (Discord). The GitHub integration consumes them with no schema changes.
- **The existing `services/ingestion/handlers/github.py` event shapers are correct and stable**: Shipped in earlier work; this feature does NOT modify the per-event shaping logic for `pull_request`, `push`, `issues`, `issue_comment`, `pull_request_review`, `check_run`. The handler is invoked unchanged via the existing ingestion path.
- **Trust tiers per shaper match the existing handler**: `pull_request.closed.merged → authoritative/state_change`, `pull_request.opened/reopened → inferential/signal`, `push → authoritative/signal`, `issues.opened/closed → authoritative`, `issue_comment → inferential/signal`, `pull_request_review.approved → authoritative/state_change`, other review states → `inferential/signal`, `check_run → authoritative`. No changes proposed in v1.
- **`X-GitHub-Event` header is the authoritative event-type signal**: Body fields are not consulted for event-type routing (matches GitHub's own documentation and the existing handler).
- **One Fyralis App per Fyralis deployment**: A single GitHub App registration backs the whole platform. Multiple Apps per deployment is a future-work item, not v1. (For comparison: a single Slack app powers IN-08, a single Discord app powers IN-09.)
- **Webhook downstream forwarding uses the existing post-commit pipeline**: The user input mentions "worker process that consumes the webhook queue." This refers to the existing `services/workers/post_commit_worker.py` that consumes `trigger_queue` rows produced by `services/ingestion/core.py::ingest()`. No new worker process is introduced.
- **Replay-cache is in-process LRU, not Redis**: Matches the cache posture of `services/webhooks/tenant_resolver.py::InstallationCache`. Cross-process replay protection is a future enhancement; observation-layer dedup is the structural backstop.
